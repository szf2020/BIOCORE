- ⏭️ T10 — 跳过（风险 #3 不需修）
- ✅ T11 — CommWatchdog timer 清理 → **风险 #4 部分 mitigated, 部分发现真 bug**：单实例 loss→restored 循环 N 次时, safetyTimer 在 'comm_restored' 处 clearTimeout + null 化（comm-watchdog.ts L83-86），destroy() 也 clearTimeout（L124-127），timer 不堆积；BatchController 不直接持有 watchdog（仅 onCommLoss 钩子），故"complete()/stop() 调 watchdog.stop()"在当前结构下不适用。**真 bug**：destroy() 不解绑自身在 plc EventEmitter 上的 4 个 listener（comm_loss / comm_restored / reconnecting / reconnected），同一 plc 上反复 new+destroy 会让旧实例继续响应 → 单次 comm_loss emit 触发所有遗留 setTimeout，timer 数 = N*(N+1)/2，且会触发 MaxListenersExceededWarning。提交为 4 用例套件（commit 59944ec，3 PASS + 1 it.fails 锁定 bug）：N 次循环不堆积 timer、destroy 清活跃 timer、重复 comm_loss 不叠 timer、跨实例 destroy/create bug demo。修复建议：destroy() 中 `this.plc.removeAllListeners('comm_loss')` 等 4 个事件（或保存 bound handler 引用做精确 off）。

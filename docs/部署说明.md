# BIOCore 部署说明

## 1. 环境要求

- **Node.js** ≥ 20 LTS
- **pnpm** ≥ 8 (通过 corepack 启用)
- **InfluxDB** 2.7+ (本机或远程,见 `.env` 配置)
- **Ollama** (可选,用于本地 LLM)
- **PLC** (S7-200 SMART G2 或兼容设备,生产模式必须)

## 2. 安装与初始化

```bash
git clone <repo>
cd biocore
corepack pnpm install
cp .env.example .env  # 根据下方说明配置
mkdir -p packages/server/data
```

## 3. 关键环境变量

| 变量 | 开发默认 | 生产必须 | 说明 |
|---|---|---|---|
| `INFLUX_URL` | http://localhost:8086 | 真实 InfluxDB URL | 时序数据库连接 |
| `INFLUX_TOKEN` | (空) | **必填** | InfluxDB All-Access Token |
| `INFLUX_ORG` | BIOCore | 改为生产 org | |
| `INFLUX_BUCKET` | BIOCore_Data | 改为生产 bucket | |
| `JWT_SECRET` | biocore-dev-secret-... | **必填强随机** | 生成: `openssl rand -base64 32` |
| `AUTH_ENABLED` | true | true | 生产必须 true,否则无 token 自动 admin |
| `MOCK_PLC` | **true** | **false 或删除** | true 时所有 PLC 读取返回模拟值 |
| `API_V0_DEPRECATION_DATE` | 启动日+180 天 | 自定义 | 旧 /api/* 路径下线日期,过此日期可移除 v0 双挂载 |

## 4. 启动顺序

```bash
# 1. 确保 InfluxDB 已运行
curl http://localhost:8086/health  # 期望 200 OK

# 2. 启动 server (会自动跑 schema migrations)
cd packages/server
corepack pnpm exec tsx watch src/index.ts

# 3. 启动 web-ui
cd packages/web-ui
corepack pnpm exec next dev --port 3000
```

启动日志关键检查项:
- `[INFO] [Influx] 已连接 ...` — InfluxDB 连接 OK
- `[Migrator] 所有 N 个 migration 已是最新` — schema 同步 OK
- `[INFO] [API] V0 兼容期截止日期: ...` — 旧路径下线日期
- **如果看到 `⚠ MOCK_PLC=true 模式启用` 警告框,生产环境必须立即停服改 .env**

## 5. 生产部署 Checklist

### 5.1 MOCK_PLC 必须禁用
```bash
# .env 中改为
MOCK_PLC=false
# 或直接删除该行
```
开机后启动日志中**不应**出现红色警告框。

### 5.2 PLC 实机连接
- 通过 `/settings/plc-config` 添加真实 PLC 连接(S7 协议 + IP + Rack/Slot)
- 注意 `SetConnectionType(3)` 必须设置(代码已自动)
- 通过"测试连接"按钮验证可达
- 启动心跳监控

### 5.3 默认 admin 密码
首次启动会自动创建 `admin / admin123` 用户。**生产部署后必须**:
1. 用 admin/admin123 登录
2. 进入 `/settings/users` 修改 admin 密码为强密码
3. 创建其他用户分配角色 (engineer/operator/viewer)

### 5.4 API Key 创建
为 MES 等外部系统创建专用 API Key:
1. 登录后进入 `/settings/api-keys`
2. 点击"创建 API Key",填名称(例: `mes-prod`)
3. **立即复制 raw key**(只显示一次),保存到 MES 系统的环境变量
4. MES 调用 biocore 时:
   ```
   curl -H "X-API-Key: ak_xxx.xxxxx" http://biocore-host:3001/api/v1/reactors
   ```

### 5.5 数据备份
SQLite 数据库位于 `packages/server/data/biocore.db`。建议:
- 通过 `/settings/data-maintenance` 手动备份
- 或定期 `cp data/biocore.db data/backups/$(date +%Y%m%d).db`

### 5.6 反向代理 + TLS
生产环境应通过 nginx/Caddy 反向代理:
```nginx
# 例: Caddy
biocore.example.com {
  reverse_proxy /api/* localhost:3001
  reverse_proxy /ws localhost:3001
  reverse_proxy localhost:3000
}
```
WebSocket 协议使用 `wss://` (Caddy 自动处理 TLS)。

## 6. 常见问题

### 服务启动失败 "[Migrator] 致命错误"
- 检查 `packages/server/data/biocore.db` 文件权限
- 检查 `packages/server/migrations/` 目录可读
- 删除 `data/biocore.db` 后重启可全新初始化(数据全失!)

### "MOCK_PLC=true 模式启用" 警告
- 开发/演示阶段正常,**生产必须移除**

### Swagger 文档 `/api/v1/docs` 404
- 检查 swagger-jsdoc 和 swagger-ui-express 是否安装
- 检查 `apis: [__filename]` 路径正确

### WS 鉴权失败 close 1008
- 检查 `?token=` 或 `?api_key=` 是否传入
- token 是否过期(JWT 默认 24h)
- API Key 是否被撤销

### 旧 /api/* 路径仍能调用
- 这是正常的 6 个月兼容期,响应头会有 `Deprecation: version="v0", sunset="<日期>"`
- 兼容期到期后,在 `server/src/index.ts` 移除 `app.use('/api', ...)` 即可下线

## 7. 硬件规格矩阵（Sprint 4 Track A 加固）

| 档次 | CPU | RAM | 磁盘 | 适用场景 | OOM 阈值（auto = RAM × 20%）|
|---|---|---|---|---|---|
| **入门** | 4 核 | 8 GB | 256 GB SSD | 1-2 反应器，无本地 AI | ~1.6 GB |
| **标准** | 8 核 | 16 GB | 512 GB SSD | 多反应器 + 本地 Ollama AI | ~3.2 GB |
| **企业** | 16+ 核 | 32 GB+ | 1 TB+ NVMe | 多车间 / 中央汇总 | ~6.4 GB |

OOM 阈值由 `BIOCORE_OOM_THRESHOLD_MB` 控制：
- `auto`（默认）= 物理 RAM × 20%
- 也可填具体 MB 数（如 `1638`）覆盖

进程稳态目标：BIOCore 内核（不含 InfluxDB / Ollama）= 200-400 MB RSS。

## 8. 运维加固环境变量（追加到 .env）

| 变量 | 默认 | 说明 |
|---|---|---|
| `NODE_OPTIONS` | `--max-old-space-size=2048` | V8 堆上限（生产建议 2-4GB）|
| `BIOCORE_OOM_THRESHOLD_MB` | `auto` | OOM watchdog 阈值；超过 graceSamples 次后 SIGTERM（生产模式）|
| `BIOCORE_OOM_GRACE_SAMPLES` | `3` | 连续 N 次采样超阈值才触发，避免瞬时尖刺误判 |
| `BIOCORE_DIAGNOSTIC_DUMP_DIR` | `./crashes` | 崩溃诊断包目录（Docker 中映射到 host volume）|
| `BIOCORE_DIAGNOSTIC_KEEP_LAST` | `50` | 保留最近 N 个诊断包，超出删最早 |
| `BIOCORE_EVENT_BUFFER_SIZE` | `1000` | SSE Last-Event-ID 重放 buffer 上限 |
| `BIOCORE_SSE_MAX_CLIENTS` | `100` | /api/v1/events 并发连接上限（防 FD 耗尽）|
| `BIOCORE_NOTIFIER_THROTTLE_MIN` | `5` | 通知防抖窗口（分钟）|
| `BIOCORE_METRICS_REQUIRE_AUTH` | `false` | `/api/v1/admin/metrics` 是否要求 admin（默认开放给 Prometheus 标准 scrape）|

## 9. Docker Compose 装机（推荐）

### 9.1 一键启动

```bash
git clone <repo>
cd biocore
cp .env.example .env
# 编辑 .env：JWT_SECRET、INFLUX_TOKEN、MOCK_PLC=false 等

docker compose up -d
docker compose ps          # 应看到 biocore-server + influxdb 都 healthy
docker compose logs -f biocore-server | grep -E "runtime-guard|notifier"
# 期望看到：
#   [runtime-guard] installed (oom_threshold_mb=...)
#   [notifier] AlertRouter ready (channels=0)
```

### 9.2 启动可选 observability profile

```bash
docker compose -f docker-compose.yml -f docker-compose.observability.yml \
  --profile observability up -d
# Prometheus: http://localhost:9090
# Grafana:    http://localhost:3002 (admin/biocore_admin)
```

### 9.3 升级流程

```bash
docker compose pull
docker compose up -d biocore-server
# 等 docker compose ps 显示 'healthy' 即升级完成
docker compose logs --tail 50 biocore-server | grep -E "error|warn" | head -10
```

健康度自检：
```bash
curl -s http://localhost:3001/api/v1/admin/health/liveness
# 期望: {"status":"ok"}（被 v1ResponseWrapper 包了一层 {code,msg,data:{status:"ok"}}）
```

### 9.4 Volume 挂载点

| Host | 容器 | 用途 |
|---|---|---|
| `./data` | `/app/data` | SQLite 数据库（必须备份）|
| `./crashes` | `/app/crashes` | 运行时诊断包 |
| `./logs` | `/app/logs` | 备用日志目录（实际日志走 docker logging driver）|
| `influxdb_data` (named) | `/var/lib/influxdb2` | InfluxDB 时序数据 |

## 10. Windows 工控机 NSSM 装机

适用于实验室现场无 Docker 的 Windows 工控机。

### 10.1 前置依赖

```powershell
# 装 Node.js 20 LTS (从 nodejs.org 或 winget)
winget install OpenJS.NodeJS.LTS

# 装 NSSM
choco install nssm
# 或从 https://nssm.cc/download 下载并放入 PATH

# 装 pnpm
corepack enable

# 启动 InfluxDB（Docker Desktop 或单独 .msi 安装）
```

### 10.2 build + 注册服务

```powershell
cd C:\biocore
pnpm install
pnpm -r build

# 以管理员身份运行
.\scripts\install-windows-service.ps1 -AppRoot "C:\biocore"

# 验证
Get-Service BioCore
nssm status BioCore
```

`install-windows-service.ps1` 会自动：
- 注册 `BioCore` Windows 服务
- 异常退出 5s 自动重启
- stdout/stderr 写到 `logs/` + 50 MB 轮转
- 注入加固 env vars（NODE_OPTIONS / BIOCORE_OOM_* / NODE_ENV=production）
- 创建 `data/`, `crashes/`, `logs/` 目录

### 10.3 卸载

```powershell
.\scripts\uninstall-windows-service.ps1
# 数据/日志/崩溃包默认保留；加 -KeepData $false 可手动指引删除
```

### 10.4 升级流程

```powershell
nssm stop BioCore
git pull
pnpm install
pnpm -r build
nssm start BioCore
```

## 11. 可观测端点（Sprint 4 Track A 新增）

| 端点 | 用途 | 鉴权 |
|---|---|---|
| `GET /api/v1/admin/health/liveness` | docker healthcheck 探针 | 公开（不需 token）|
| `GET /api/v1/admin/health` | JSON 完整快照（内存/句柄/事件循环/PLC 等）| admin |
| `GET /api/v1/admin/health/timeseries` | 最近 24h ring buffer（1440 点）| admin |
| `GET /api/v1/admin/metrics` | Prometheus 文本格式 | 默认公开（可锁）|
| `GET /api/v1/admin/crashes` | 列出崩溃诊断包 | admin |
| `GET /api/v1/admin/crashes/:name` | 下载单个诊断包 JSON | admin |
| `GET /api/v1/events` | SSE 事件流（process_restart / oom_threshold / plc_disconnect_5min / uncaught_exception / heap_growth_anomaly）| API Key |
| `GET /api/v1/notifications/channels` | 通道列表 | admin |
| `PUT /api/v1/notifications/channels/:id` | 创建/更新通道 | admin |
| `DELETE /api/v1/notifications/channels/:id` | 删除通道 | admin |
| `POST /api/v1/notifications/channels/:id/test` | 触发测试通知 | admin |
| `GET /api/v1/notifications/rules` | 规则列表 + 可用事件类型 | admin |
| `PUT /api/v1/notifications/rules` | 替换全量规则 | admin |

前端入口：
- `/admin/health` — 实时健康度看板（10s 自动刷新）
- `/settings/notifications` — 通道+规则管理

## 产品定位与差异化宣言

**产品名称：** BIOCore  
**版本代号：** v0.1 
**目标市场：** 非GMP实验室R&D环境，面向高校、科研院所及生物科技企业的工艺开发部门  
**核心主张：** 以竞品1/5的软件成本和无限的二次开发自由度，提供覆盖发酵全生命周期的智能控制与数据分析平台

### 1.2 竞品核心痛点分析

**痛点一：硬件深度锁定**

Eppendorf TruBio只能搭配BioFlo系列控制器使用，一台BioFlo 320基础配置（不含罐体）售价在$25,000-40,000区间。Sartorius的Biobrain Supervise虽然通过OPC UA支持第三方设备，但原生连接仍以自家Biostat系列为主。这意味着研究团队一旦选定某品牌的控制软件，就被锁定在该品牌的硬件生态中，后续的耗材、维修和升级都缺乏议价空间。

对于预算有限的高校实验室或初创生物技术公司，自行组装发酵罐（采购标准玻璃罐体 + 通用传感器 + 蠕动泵 + PLC）的成本可以控制在$5,000-8,000，仅为品牌整机的1/5。但这些自组装设备目前缺乏一款专业的控制软件来统合管理。

**痛点二：封闭的数据生态**

三款竞品的数据导出能力均有不同程度的限制。TruBio的数据以专有格式存储，虽然支持CSV导出但丢失了元数据上下文。Lucullus使用Oracle数据库，数据可靠性极高，但Oracle的许可费用和运维复杂度对小团队而言是沉重负担。在AI/ML驱动的生物工艺优化成为大趋势的今天，这些封闭的数据体系严重阻碍了研究人员将发酵数据接入Python/R数据科学工具链的效率。

**痛点三：零AI原生能力**

截至目前，三款竞品均没有提供AI原生集成。研究人员需要手动导出数据，在Jupyter Notebook中完成清洗和分析，再将结论手动反馈回工艺参数。BIOCore通过内置的AI API网关，让研究人员可以直接在发酵监控界面中向大语言模型提问："当前批次的DO曲线与历史最优批次相比是否存在异常？"——这种体验在现有竞品中完全不存在。

### 1.3 目标用户画像

| 用户角色 | 核心需求 | 当前痛点 |
|---------|---------|---------|
| 高校发酵实验室PI/导师 | 低成本搭建多套并行发酵平台，支持学生快速上手 | 品牌设备预算不足，开源方案缺乏专业控制逻辑 |
| 生物技术初创公司工艺工程师 | 快速迭代菌株筛选工艺，积累可追溯的批次数据 | 买不起Sartorius全套方案，数据散落在Excel中 |

### 2.3 应用逻辑层与数据库架构概览

应用逻辑层基于Node.js构建，采用monorepo结构，包含7个核心包：

| 包名 | 职责 | 关键技术 |
|------|------|---------|
| `plc-driver` | 多协议通讯(S7+Modbus RTU)、双向心跳(VB400/VB401)、安全报警 | node-snap7库(S7协议)、modbus-serial库(Modbus RTU/TCP)、V区地址映射 |
| `batch-engine` | ISA-88状态机、配方解析、Phase执行 | XState v5、JSON配方 |
| `data-service` | 数据采集聚合、审计日志、批次报表 | InfluxDB + SQLite双库 |
| `ai-local` | 本地LLM交互、自然语言查询、批次摘要 | Ollama + Qwen2.5-7B |
| `ai-analytics` | CUSUM异常检测、DTW批次匹配、包络线 | 纯JS统计算法 |
| `soft-sensor` | ONNX模型推断、特征工程、模型管理 | onnxruntime-node |
| `web-ui` | React前端、Dashboard、配方编辑器 | Next.js + Plotly.js |

**双数据库架构：**

| 数据库 | 存储内容 | 写入频率 | 技术选型理由 |
|-------|---------|---------|------------|
| **InfluxDB OSS 2.x** | 过程数据（温度/pH/DO/转速/流量等）、软测量估算值、离线取样数据 | 每分钟1次 | 列式压缩效率高10倍；Flux语言原生支持时间窗口聚合和多批次对比 |
| **SQLite** | 批次记录、配方版本、状态机流转、审计追踪、报警历史、用户管理、AI会话 | 事件驱动 | 零运维单文件部署；WAL模式；触发器禁止UPDATE/DELETE确保审计不可篡改 |

详细的数据库Schema、InfluxDB Measurement设计、Retention Policy、备份策略等内容，请参阅各技术子文档中的相关章节。

## 第三部分：开发路线图

### Phase 1 — MVP核心控制（第1-3个月）

**目标：** 实现基本的发酵过程监控与控制，可替代手动操作。

|-------|-------|---------|
| M1.1 PLC通讯 | plc-driver模块(node-snap7+modbus-serial) | 稳定读写V区变量；双向心跳(VB400/VB401)3秒超时自动Hold；支持S7+Modbus双协议 |
| M1.2 PID控制 | PLC程序 | 温度控制精度±0.3°C，pH控制精度±0.05，DO四种策略可切换且级联正常 |
| M1.3 安全连锁 | PLC安全逻辑 | 8个PID回路的安全连锁 + 蒸汽/冷却互锁，全部通过模拟故障注入测试 |
| M1.4 基础UI | Web监控面板 | 实时趋势图、参数设定、手动/自动模式切换 |
| M1.5 数据记录 | data-service模块 | InfluxDB每分钟采样持续写入 + SQLite审计日志，72小时连续运行无数据丢失 |

### Phase 2 — 批次引擎与配方（第4-6个月）

**目标：** 实现ISA-88状态机和JSON配方管理，支持SIP/CIP自动化。

| 里程碑 | 交付物 | 验收标准 |
|-------|-------|---------|
| M2.1 状态机 | batch-engine(XState v5) + CommWatchdog | 6个S88状态完整流转，含comm_loss通讯断线自动Hold + comm_restored恢复等待人工确认 |

## 第五部分：技术风险与缓解策略

|---------|---------|------|------|---------|
| R1 | Node.js单线程事件循环在高负载下延迟，影响状态机确定性 | 中 | 高 | PLC层保留独立安全逻辑作为兜底；Node.js端状态机采用独立Worker Thread隔离；设定200ms最大允许延迟阈值 |
| R2 | node-snap7依赖Snap7 C库需native编译，部分平台可能编译失败 | 低 | 高 | 提供预编译binary；备选modbus-serial纯JS方案；变频器已通过modbus-serial独立支持Modbus RTU |
| R3 | 用户自组装发酵罐的传感器信号质量参差不齐 | 高 | 中 | 提供详细的传感器选型指南和接线规范；软件内置信号质量诊断（检测断线、饱和、噪声水平） |
| R4 | 本地LLM(7B)在CPU模式下推理速度慢，用户体验不佳 | 中 | 中 | 自然语言查询和批次摘要均为非实时场景，30-60秒等待可接受；推荐用户配置≥32GB RAM；提供Q4/Q5/Q8多种量化选项权衡速度与质量 |
| R5 | 本地软测量模型训练数据不足(<10批次)导致精度差 | 高 | 中 | 提供基于公开发酵数据集预训练的通用基线模型；UI明确显示模型R²和训练样本量；R²<0.85时禁止上线推断并提示用户积累更多数据 |
| R6 | AI建议被操作员误当成确定性指令执行，导致工艺偏差 | 中 | 高 | 架构层面强制"建议缓冲区"模式——AI永远不能直接写入PLC；所有AI输出在UI上以虚线/灰色/警告标识明确区分；"采纳"操作写入审计日志 |
| R7 | 通讯断线期间工艺损失风险 | 中 | 高 | 双向心跳3秒检测；PLC独立维持温度/pH；Hold后等待人工确认恢复；超1小时安全停机 |
| R8 | 竞品厂商通过固件升级封锁第三方通讯接口 | 低 | 中 | 首期产品定位于自组装设备市场，不依赖任何品牌控制器；后期通过OPC UA适配主流品牌 |

## 附录B：技术选型决策记录

| 决策点 | 选择 | 备选方案 | 选择理由 |
|-------|------|---------|---------|
| PLC型号 | S7-200 SMART G2 | S7-1200, Arduino+工业扩展 | G2性价比最优，工业级可靠性，目标市场(中国/东南亚)货源充足且价格低廉 |
| 通讯协议 | S7协议(node-snap7, 首选) + Modbus RTU/TCP(modbus-serial) | OPC UA, MQTT, nodes7 | node-snap7基于Snap7 C库，工业级成熟，支持S7全系列；modbus-serial用于变频器RS232通讯(npm周下载10万+)；弃用nodes7因维护不活跃 |
| 后端运行时 | Node.js (v20 LTS) | Python (FastAPI), Go | Node.js的异步I/O天然适合实时数据流；npm生态中S7/Modbus库成熟；前后端同语言降低团队门槛 |
| 状态机库 | XState v5 | node-red-contrib-state-machine, 自研 | XState有完整的可视化工具链，支持持久化和快照恢复，社区活跃度远高于Node-RED FSM插件 |
| 前端框架 | React + Next.js | Vue, Svelte | React生态最大，UI组件库(shadcn/ui)成熟，Next.js支持SSR便于嵌入式设备的Web HMI |
| 趋势图库 | Plotly.js | Chart.js, D3.js | Plotly原生支持时序数据、多Y轴、框选缩放、数据点悬浮——这些都是发酵监控的刚需 |
| 数据库 | InfluxDB OSS 2.x（时序）+ SQLite（业务） | PostgreSQL, MySQL, TimescaleDB | InfluxDB的列式压缩对时序数据存储效率高10倍，Flux语言原生支持时间窗口聚合和多批次对比查询；SQLite零运维单文件部署，对实验室用户最友好；双库分工明确，避免关系型数据库处理时序数据的性能瓶颈 |
| AI接口 | 本地优先：Ollama + Qwen2.5-7B（LLM）、onnxruntime-node（ML推断）、纯JS统计算法（异常检测） | 云端API(Anthropic/OpenAI)、TensorFlow.js、Python Flask微服务 | 本地优先确保断网可用和数据不出境；Ollama的OpenAI兼容API让同一套代码无缝切换本地/云端；onnxruntime-node无需Python运行时即可推理ML模型；CUSUM/DTW等统计算法用纯JS实现零外部依赖 |

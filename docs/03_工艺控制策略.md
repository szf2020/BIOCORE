本节是BIOCore的工艺核心——描述软件如何在发酵全生命周期中实现精细化的过程控制。与简单的PID闭环不同，真正的发酵控制需要理解微生物代谢的内在逻辑：细胞生长消耗底物→底物消耗产生耗氧需求→耗氧改变溶氧水平→代谢副产物改变pH→pH偏移反过来影响细胞代谢。这些变量不是独立的，而是一条环环相扣的因果链。BIOCore的控制策略正是围绕这条因果链来设计的。

#### 2.6.1 软件实时测算参数体系（Software-Calculated Parameters）

发酵过程中许多关键参数无法被传感器直接测量，但可以通过已有的在线信号经过数学运算实时推导。这些"软件测算值"构成了高级控制策略的数据基础，也是BIOCore区别于简单SCADA系统的核心能力。

**基于现有P&ID传感器信号可实时计算的参数：**

| 测算参数 | 符号 | 物理含义 | 计算公式/方法 | 输入信号来源 | 更新频率 |
|---------|------|---------|-------------|-----------|---------|
| 氧消耗速率 | OUR | 细胞每小时消耗的氧气量(mmol/L/h) | OUR = kLa × (DO* - DO) | AI-3(DO实测), AI-5(通气量), 搅拌转速(VFD) → kLa估算 | 每分钟 |
| 二氧化碳释放速率 | CER | 细胞每小时释放的CO₂量(mmol/L/h) | 需尾气CO₂分析仪（AI-7预留口） | AI-7(预留，如配备尾气分析仪) | 每分钟 |
| 呼吸商 | RQ | CO₂产生/O₂消耗的比值，判断代谢类型 | RQ = CER / OUR | 由OUR和CER导出 | 每分钟 |
| 比生长速率 | μ | 细胞群体的瞬时增殖速率(1/h) | 由OUR变化率推算：μ ≈ d(ln(OUR))/dt | OUR的时间导数 | 每10分钟（滑窗回归） |
| 累积补料量 | V_feed | 已泵入罐内的补料母液总体积(mL) | V_feed = Σ(泵速 × 时间步长) | Q0.1 PWM占空比 × 泵管流速系数 | 每分钟累积 |
| 累积补碱量 | V_base | 已泵入的碱液总体积(mL) | V_base = Σ(碱泵速 × 时间步长) | Q0.0 PWM占空比 × 泵管流速系数 | 每分钟累积 |
| 累积耗氧量 | O₂_total | 发酵全程的总氧消耗(mmol) | O₂_total = Σ(OUR × Δt × V_liquid) | OUR × 液相体积 × 时间积分 | 每分钟累积 |
| 体积传质系数 | kLa | 反应器当前的氧传递能力(1/h) | kLa = f(N, Vs, μL) 经验关联式，或动态法实测 | 搅拌转速N(VFD), 表观气速Vs(AI-5), 液相体积 | 每10分钟 |
| 表观气速 | Vs | 空气通过液面的线速度(m/s) | Vs = Q_air / A_cross | AI-5(空气流量) / 罐体截面积(已知) | 每分钟 |
| F₀灭菌值 | F₀ | SIP阶段的累积杀菌效力(min) | F₀ = Σ 10^((T-121)/z) × Δt, z=10°C | AI-0(罐温), 每分钟积分 | SIP阶段每分钟 |
| 罐内液相体积 | V_liquid | 当前罐内实际液体体积(L) | V_liquid = V_initial + V_feed + V_base + V_acid - V_sample | 称重(AI-6)交叉验证或泵速积分 | 每分钟 |
| 比耗氧速率 | qO₂ | 单位生物量的耗氧速率 | qO₂ = OUR / X（X为生物量浓度） | OUR / 软测量估算OD(v2.0) | 每分钟(v2.0) |

**kLa经验关联式的软件实现：**

kLa（体积传质系数）是溶氧控制策略的核心参数，它决定了当前反应器能向培养液中"泵入"多少氧气。BIOCore内置了Van't Riet经验关联式并支持用户自定义系数：

```javascript
// soft-sensor/kla-estimator.js

/**
 * Van't Riet kLa经验关联式
 * kLa = C × (P/V)^a × Vs^b
 *
 * 其中:
 *   P/V = 单位体积搅拌功率 (W/L)
 *   Vs  = 表观气速 (m/s)
 *   C, a, b = 经验系数（依罐体几何和搅拌桨类型而定）
 *
 * 对5L台式罐的典型值:
 *   C = 0.026, a = 0.4, b = 0.5 (Rushton涡轮桨, 水性体系)
 */
function estimateKLa(rpm, airflowNLmin, liquidVolumeL, config) {
  const { C = 0.026, a = 0.4, b = 0.5, impellerDiameter = 0.05, Np = 5.0 } = config;

  // 搅拌功率估算: P = Np × ρ × N³ × D⁵
  const N = rpm / 60;  // rps
  const rho = 1000;    // 水性培养基密度 kg/m³
  const P = Np * rho * Math.pow(N, 3) * Math.pow(impellerDiameter, 5);  // W
  const PV = P / (liquidVolumeL / 1000);  // W/m³

  // 表观气速: Vs = Q / A
  const Q = (airflowNLmin / 1000) / 60;  // m³/s (NL/MIN → m³/s)
  const tankDiameter = 0.15;  // 5L罐内径约150mm
  const A = Math.PI * Math.pow(tankDiameter / 2, 2);  // m²
  const Vs = Q / A;  // m/s

  const kLa = C * Math.pow(PV, a) * Math.pow(Vs, b) * 3600;  // 1/h
  return { kLa, PV, Vs };
}
```

**OUR的动态计算方法：**

在没有尾气分析仪的情况下（本系统AI-7为预留口），OUR可以通过稳态物料平衡法近似计算：

```
OUR = kLa × (DO* - DO_measured)

其中:
  kLa  = 由搅拌转速和通气量通过经验关联式估算（见上）
  DO*  = 饱和溶氧值（取决于温度、压力、培养基组成）
       = DO_100% × (P_tank / P_atm) × Henry系数温度校正
  DO_measured = AI-3 DO传感器实测值

更精确的方法（当配备尾气分析仪时）:
  OUR = (F_in × y_O2_in - F_out × y_O2_out) / V_liquid
  其中 F_in = 进气摩尔流量, y_O2_in = 0.2095(空气), y_O2_out = 尾气O₂摩尔分数
```

这些测算参数全部存入InfluxDB的独立measurement `calculated_params`，与原始传感器数据分开存储但共享相同的batch_id和时间轴，便于前端同屏展示。

#### 2.6.2 溶氧控制的四种可选策略

溶氧（DO）控制是好氧发酵中最复杂的控制问题。不同的菌株、不同的工艺阶段、不同的实验目的，需要完全不同的DO控制哲学。BIOCore不预设"唯一正确"的DO策略，而是提供四种可选策略，由工艺工程师在配方JSON中为每个Phase独立选择。

**四种策略的本质区别：**

```
                         DO是否被闭环控制？
                        /                \
                      是                  否
                    /    \              /    \
           操纵变量是什么？        哪个变量恒定？
              /      \              /        \
         氧供给侧    补料侧      氧供给恒定   补料恒定
            |          |            |           |
        策略一       策略二       策略三       策略四
      主动调氧     主动调补料     恒定氧气     恒定补料
```

---

**策略一：主动调节氧气（Active O₂ Control）**

**原理：** DO作为被控变量，搅拌转速和通气量作为操纵变量。补料速率按照预设的配方曲线（指数/恒速）独立运行，不受DO影响。这是最经典的发酵DO控制策略，适用于大多数好氧发酵场景。

**控制逻辑：**

```
DO PID控制器
  输入(PV): AI-3 DO实测值
  设定值(SV): 配方设定（如30%）
  
  Level 1 — 搅拌级联（优先）
    PID输出 → 搅拌转速SV修正量 → VFD RS232写入
    范围: 200rpm(最低) ~ 1200rpm(上限)
    作用: 提高转速 → 增大气泡剪切 → 提高kLa → 提升DO
  
  Level 2 — 通气级联（搅拌达上限后激活）
    当搅拌≥1200rpm 且 DO仍<SV:
    PID输出 → AO-2 空气流量调节阀
    范围: 1 NL/MIN ~ 30 NL/MIN
    AI-5 空气流量计提供闭环反馈

  前馈增强（可选开启）:
    当补料速率发生变更时，前馈控制器提前计算:
    ΔF_feed → 预期ΔOUR → 需要的ΔkLa → 提前调高搅拌/通气
    （详见下方"前馈控制器"）

  极限保护:
    搅拌和通气均达上限且DO<10%持续5min → 触发Hold + 报警
```

**前馈控制器——补料量是未来耗氧需求的领先指标：**

从补料泵加速到OUR上升之间有5-15分钟的代谢滞后。策略一利用这个时间差实现"主动预判"：

```
t=0min    补料泵速率↑
  │
  ├──→ 前馈控制器立即计算:
  │    ΔF_glucose = ΔF_feed × C_feed / MW_glucose    (mmol/h)
  │    ΔOUR = ΔF_glucose × Y_O2/S / V_liquid          (mmol/L/h)
  │    ΔkLa_needed = ΔOUR / (DO* - DO_target)          (1/h)
  │    → 反算出搅拌/通气增量 → 立即下发至PLC
  │
  ▼
t=8-15min OUR如预期上升，但DO几乎没有下降
          反馈PID只需做微小修正
```

**适用场景：** 大多数好氧发酵的默认策略。适合E.coli、酵母等需要维持充足氧供给的高密度培养。工艺工程师关注的核心指标是产量最大化，愿意让搅拌和通气自动跟随耗氧需求变化。

**配方JSON配置示例：**

```json
{
  "DO_strategy": "active_O2",
  "DO_sv": 30,
  "cascade_level1": {"actuator": "agitation", "range_rpm": [200, 1200]},
  "cascade_level2": {"actuator": "airflow", "range_NL_min": [1, 30]},
  "feedforward_enabled": true,
  "feedforward_delay_min": 10
}
```

**策略二：主动调节补料（Active Feed Control / DO-stat）**

**原理：** DO作为被控变量，补料速率作为操纵变量。搅拌和通气保持在用户设定的固定值。当DO下降时，系统通过降低补料速率来减少耗氧需求；当DO回升时，系统加大补料速率。本质上是用DO反馈来自适应地控制细胞的代谢速率。

**控制逻辑：**

```
DO → 补料速率 PID控制器
  输入(PV): AI-3 DO实测值
  设定值(SV): 配方设定（如30%）
  输出(CV): 补料泵P02(Q0.1) PWM转速

  搅拌转速: 固定值（由配方设定，如800rpm）
  通气量: 固定值（由配方设定，如15 NL/MIN）

  PID方向: 反向作用（DO↓ → 补料速率↓, DO↑ → 补料速率↑）

  安全下限: 补料速率不低于维持量 F_maintenance
    F_maintenance = m_s × X × V_liquid
    其中 m_s = 维持系数(约0.04 g/g/h), X = 估算生物量
    作用: 确保细胞不会因补料完全停止而死亡

  安全上限: 补料速率不超过临界值 F_critical
    F_critical = μ_crit × X × V_liquid / (Y_xs × S_feed)
    作用: 防止补料速率超过μ_max导致的溢流代谢
```

**策略二的核心优势——天然防溢流：**

```
传统指数补料(策略四)的风险:
  预设μ=0.15/h → 但实际菌体μ_max可能只有0.12/h
  → 底物积累 → 溢流代谢(Crabtree效应) → 乙酸/乙醇积累 → 抑制生长

策略二的自调节机制:
  补料过快 → OUR↑ → DO↓ → PID自动降低补料速率
  补料过慢 → OUR↓ → DO↑ → PID自动提高补料速率
  → 补料速率自动追踪细胞的实际代谢能力
  → 天然避免溢流代谢，无需精确知道μ_max
```

**适用场景：** 菌株代谢特性尚不清楚（新菌株首次发酵、基因改造后的菌株）；需要避免溢流代谢的保守策略；酿酒酵母（S. cerevisiae）的Crabtree效应敏感型发酵；不要求最大化生长速率，优先保证过程安全性的场景。

**配方JSON配置示例：**

```json
{
  "DO_strategy": "active_feed",
  "DO_sv": 30,
  "agitation_fixed_rpm": 800,
  "airflow_fixed_NL_min": 15,
  "feed_pump": "P02",
  "feed_pid_direction": "reverse",
  "feed_min_mL_h": 0.5,
  "feed_max_mL_h": 25.0
}
```

**策略三：恒定氧气供给（Constant O₂ Supply）**

**原理：** 搅拌转速和通气量均保持在用户设定的固定值，DO不进行闭环控制——它自然浮动，反映细胞的实时耗氧状态。补料速率按照预设配方曲线运行。DO被视为一个**观测变量**而非控制变量。

**控制逻辑：**

```
搅拌转速: 固定值（由配方设定，如600rpm）
通气量: 固定值（由配方设定，如 10 NL/MIN）
DO: 仅监测和记录，不闭环控制

补料: 按配方设定的指数/恒速/脉冲策略独立运行

DO仅用于:
  ① 实时显示在Dashboard趋势图上
  ② OUR软件测算的输入参数（OUR = kLa × (DO* - DO)，此时kLa为常数）
  ③ 触发关键事件检测（DO Spike → 碳源耗尽信号）
  ④ 安全底线报警（DO<5%持续5min → 软报警提示，非自动干预）
```

**策略三的价值——当DO本身就是你要研究的对象：**

| 应用场景 | 为什么选策略三 |
|---------|-------------|
| 微需氧/厌氧发酵 | 某些产物（如丁醇、乙醇）在低氧或无氧条件下产率更高，DO不应被强制拉高 |
| kLa标定实验 | 在已知搅拌和通气条件下观察DO的自然响应，用于实测kLa值 |
| 菌株耗氧特性表征 | 固定供氧条件下观察DO曲线形态，表征不同菌株/培养基的氧需求差异 |
| 低密度预培养 | Batch阶段初期细胞密度低，耗氧远低于供氧能力，闭环控制无意义 |
| 缩小成本/简化系统 | 无需配置DO比例调节阀，蠕动泵不参与DO控制，降低硬件和调试成本 |

**配方JSON配置示例：**

```json
{
  "DO_strategy": "constant_O2",
  "agitation_fixed_rpm": 600,
  "airflow_fixed_NL_min": 10,
  "DO_monitor_only": true,
  "DO_alarm_low": 5,
  "DO_alarm_low_delay_min": 5
}
```

**策略四：恒定补料速度（Constant Feed Rate）**

**原理：** 补料速率按照预设曲线严格执行（恒速或预计算的指数曲线），不受任何在线反馈信号的调整。搅拌和通气同样维持固定值。这是最简单的开环控制模式——所有参数在配方中预定义，系统只负责精确执行，不做自适应调节。

**控制逻辑：**

```
补料速率: 严格按配方执行
  恒速模式: F = F_constant (如 5 mL/h)
  指数模式: F(t) = F₀ × e^(μ_set × t)
  分段模式: 按时间表切换速率 [{0-4h: 2mL/h}, {4-8h: 5mL/h}, {8-24h: 10mL/h}]

搅拌转速: 固定值
通气量: 固定值
DO: 仅监测，不控制

所有PID控制:
  温度PID: 正常运行（温度始终需要闭环控制）
  pH PID: 正常运行（pH始终需要闭环控制）
  DO PID: 禁用
  搅拌PID: 禁用（VFD直接接收固定频率设定）
```

**策略四的价值——可重复性优先于最优性：**

| 应用场景 | 为什么选策略四 |
|---------|-------------|
| 工艺对标实验 | 需要严格复现文献中的操作条件，任何自适应调节都会引入不可控变量 |
| DoE实验设计 | 正在做补料速率的单因素或多因素实验，补料速率必须是精确的自变量 |
| 批次间一致性验证 | 验证罐-罐一致性（如从F01复现到F02），需要所有操作条件完全固定 |
| 教学演示 | 学生首次操作，减少自动控制的复杂性，便于理解基础发酵动力学 |
| 厌氧/微需氧发酵 | 某些发酵无需供氧控制，固定低通气或不通气即可 |

**配方JSON配置示例：**

```json
{
  "DO_strategy": "constant_feed",
  "agitation_fixed_rpm": 500,
  "airflow_fixed_NL_min": 5,
  "DO_monitor_only": true,
  "feed_profile": {
    "mode": "exponential",
    "F0_mL_h": 2.0,
    "mu_set": 0.15,
    "max_mL_h": 25.0
  }
}
```

| 补料速率由谁决定 | 配方预设曲线 | DO PID自动调节 | 配方预设曲线 | 配方预设曲线 |
| 搅拌/通气由谁决定 | DO PID自动调节 | 配方预设固定值 | 配方预设固定值 | 配方预设固定值 |

| 最大生产效率 | **高** | 中（受DO约束） | 取决于预设值 | 取决于预设曲线 |
| 过程可重复性 | 中（搅拌/通气在变） | 中（补料速率在变） | **高（所有输入固定）** | **高（所有输入固定）** |
| 适合新菌株首试 | ✓ | **✓✓（最推荐）** | △ | ✗ |

**策略切换的Phase级粒度：**

#### 2.6.3 pH作为代谢状态指示器的智能控制

在实际发酵中，pH不仅仅是一个需要"保持在设定值"的参数，它的变化趋势本身就是细胞代谢状态的最灵敏指示器。BIOCore在传统的pH双向PID控制基础上，增加了**代谢状态推断层**，从pH动态中提取操作决策所需的工艺信息。

**pH动态特征与代谢状态的对应关系（内置于BIOCore的规则引擎）：**

| pH变化特征 | 代谢事件推断 | BIOCore自动响应 |
|-----------|-----------|--------------|
| pH持续下降且碱泵累积量加速增长 | 乙酸/乳酸等有机酸积累（溢流代谢） | 触发软报警"疑似溢流代谢"；如补料模式为恒速，建议降低补料速率 |
| pH突然上升且DO同步出现Spike | 碳源耗尽，细胞开始消耗有机酸作为替代碳源 | 判定为"DO Spike事件"→触发配方Phase切换（Batch→Fed-batch） |
| pH稳定但碱消耗速率(dV_base/dt)持续上升 | 细胞处于旺盛生长期，氨基酸代谢释放H⁺ | 正常状态，μ正在上升，可作为比生长速率的辅助估算 |
| pH振荡（酸泵碱泵交替频繁启停） | PID死区设置过小或混匀时间不足 | 自动加大死区±0.02并延长最小泵停间隔至30秒 |
| pH骤降>0.5 within 5min且碱泵满速仍无法回调 | 严重代谢事故或培养基组分异常 | 触发Hold状态→关闭补料泵→降搅拌→HMI弹窗要求人工干预 |

**碱消耗速率作为比生长速率的在线估算：**

在好氧发酵中，碱消耗速率与细胞生长速率之间存在近似线性关系（因为细胞生长伴随的NH₄⁺同化和有机酸产生都会消耗碱液）。BIOCore利用这一关系实现μ的在线软估算：

```javascript
// ai-analytics/growth-rate-estimator.js

class GrowthRateFromBaseConsumption {
  constructor(config) {
    this.windowMinutes = config.window || 30;  // 滑窗宽度
    this.baseHistory = [];  // [{timestamp, cumBaseVolume}]
    this.YbaseX = config.YbaseX || null;  // mL碱/g干菌体 (需从历史批次标定)
  }

  /**
   * 每分钟调用：输入累积碱消耗量，输出比生长速率估算
   */
  update(timestamp, cumBaseVolumeML, liquidVolumeL) {
    this.baseHistory.push({ timestamp, cumBaseVolumeML });

    // 保留最近N分钟的数据
    const cutoff = timestamp - this.windowMinutes * 60 * 1000;
    this.baseHistory = this.baseHistory.filter(p => p.timestamp > cutoff);

    if (this.baseHistory.length < 5) return null;  // 数据不足

    // 线性回归计算碱消耗速率 dVbase/dt (mL/h)
    const baseRate = this.linearRegression(this.baseHistory);

    // 如果有历史标定的YbaseX系数，可以估算μ
    // μ ≈ (dVbase/dt × C_base) / (YbaseX × X × V_liquid)
    // 注意：这需要v2.0软测量提供的X(生物量浓度)才能完整计算
    // v1.0阶段仅输出碱消耗速率趋势作为参考

    return {
      baseConsumptionRate: baseRate.toFixed(2),  // mL/h
      trend: baseRate > this.lastRate ? 'increasing' : 'stable',
      growthPhaseEstimate: this.classifyPhase(baseRate)
    };
  }

  classifyPhase(rate) {
    // 基于碱消耗速率的变化模式粗略判断生长阶段
    if (rate < 0.5) return 'LAG';           // 延迟期：碱消耗极低
    if (this.isAccelerating()) return 'LOG'; // 对数期：碱消耗加速
    if (this.isDecelerating()) return 'STAT'; // 稳定期：碱消耗减速
    return 'UNKNOWN';
  }
}
```

#### 2.6.4 ML增强的温度控制——自适应PID参数调整

传统PLC中的温度PID参数（Kp、Ki、Kd）在整个发酵过程中是固定的。但实际上，发酵罐的热力学特性会随发酵进程动态变化：

- **液相体积变化**：随着补料的持续加入，罐内液位从3L增长到5L，热容量增大60%，系统热惯性显著改变。
- **生物产热变化**：对数生长期的生物产热量可达保温期的5-10倍，相当于引入了一个动态的内部热源扰动。
- **培养液黏度变化**：高密度发酵后期，培养液黏度可升高2-3倍，影响夹套换热效率。

固定PID参数在发酵初期（低液位、低产热）可能调得很好，但到了高密度期（高液位、高产热、高黏度）就会出现振荡或响应迟缓。BIOCore通过两种策略应对这一问题：

**策略一：基于规则的分阶段PID参数切换（v1.0）**

在配方JSON中为不同发酵阶段预设不同的PID参数集，由batch-engine在Phase切换时自动下发：

```json
{
  "temperature_pid_profiles": {
    "SIP_heating": {
      "Kp": 8.0, "Ki": 0.5, "Kd": 2.0,
      "note": "SIP阶段需要快速升温到121°C，激进参数，允许小幅超调"
    },
    "SIP_hold": {
      "Kp": 3.0, "Ki": 0.8, "Kd": 1.0,
      "note": "121°C保温阶段，偏保守，绝对不能超调（避免焦糖化）"
    },
    "cooling_down": {
      "Kp": 6.0, "Ki": 0.3, "Kd": 1.5,
      "note": "降温阶段，冷却水响应滞后大，需加强微分抑制"
    },
    "fermentation_early": {
      "Kp": 4.0, "Ki": 0.6, "Kd": 1.0,
      "note": "发酵初期，液位低(~3L)，产热低，中等参数"
    },
    "fermentation_late": {
      "Kp": 2.5, "Ki": 0.8, "Kd": 0.5,
      "note": "发酵后期，液位高(~5L)，产热高，需降低比例增益防振荡"

```

**策略二：ML自适应PID调参（v2.0，本地运行）**

当用户积累了足够多的批次数据后，BIOCore可以训练一个轻量级模型来预测最优的PID参数：

```
训练数据构造（自动从历史批次中提取）：

对每个历史批次，每10分钟提取一组快照：
  输入特征 X:
    - 当前液相体积 V_liquid (L)
    - 当前OUR (mmol/L/h) → 代表生物产热水平
    - 当前搅拌转速 (rpm) → 影响混合均匀性和局部温度梯度
    - 蒸汽/冷却阀当前开度 (%) → 当前控制负荷
    - 罐温与设定值的偏差 e(t) → 当前误差

  目标变量 Y:
    - 该时刻之后10分钟内的温控性能指标:
      IAE = ∫|e(t)|dt  (积分绝对误差，越小越好)
      超调量 = max(PV - SV) / SV × 100%
      综合评分 = -log(IAE) - 5×超调量  (越大越好)

  对应的PID参数:
    - 当时PLC中运行的Kp, Ki, Kd值

模型训练:
  使用梯度提升树(LightGBM)学习：
    f(V_liquid, OUR, rpm, valve_pos, error) → best(Kp, Ki, Kd)

  训练完成后导出为ONNX → 每10分钟由onnx-runner.js推理 →
  推荐的PID参数与当前PLC中运行的参数对比 →
  如差异>10%则通过AISuggestionCard提示操作员确认调整
```

#### 2.6.5 补料阶段的关键控制算法

Fed-batch补料阶段是发酵工艺中产量优化的核心时段。BIOCore内置三种补料算法，在配方JSON中由用户选择：

**算法一：指数补料（Exponential Feeding）**

```
F(t) = F₀ × e^(μ_set × t)

其中:
  F₀ = (μ_set × X₀ × V₀) / (Y_xs × S_feed)  初始补料速率
  μ_set = 目标比生长速率 (1/h)，通常设为μ_max的60-80%以避免溢流
  X₀ = 补料开始时的生物量浓度 (g/L)，来自v2.0软测量或离线取样
  V₀ = 补料开始时的液相体积 (L)
  Y_xs = 底物-菌体得率系数 (g菌/g底物)
  S_feed = 补料母液底物浓度 (g/L)

Node.js层每分钟计算当前应有的泵速 → 下发至PLC Q0.1 PWM设定值
```

**算法二：DO-stat补料（基于溶氧反馈的自适应补料）**

```
原理: 当碳源耗尽时DO会出现上升（因耗氧需求突然降低）。
      利用这一特征"试探性"地补料——
      只在DO开始上升时补入一个脉冲量的底物，然后等待DO再次上升。

实现:
  当 DO > DO_setpoint + 5% 且 dDO/dt > 0.5%/min:
    → 启动补料泵以固定速率运行 pulse_duration 秒
    → 然后停泵，等待下一次DO上升
    → pulse_duration 随发酵时间逐渐增大（因生物量在增加）

优势: 天然避免过量补料导致的溢流代谢
缺陷: 生长速率不稳定，细胞反复经历"饱-饥"循环
适用: 不确定μ_max或Y_xs等动力学参数时的保守策略
```

**算法三：μ-stat补料（基于比生长速率反馈的闭环补料，v2.0）**

```
原理: 实时计算当前的比生长速率μ（通过OUR或碱消耗速率估算），
      用PID控制器调节补料速率使μ跟踪设定值μ_set。

  PID输入: μ_measured (从OUR或碱消耗速率推算)
  PID输出: 补料速率修正量 ΔF
  补料速率: F(t) = F_baseline(t) + ΔF

  F_baseline = 指数补料公式提供的理论速率
  ΔF = PID根据μ偏差进行修正

优势: 真正的闭环控制，自动适应细胞状态变化
依赖: v2.0的软测量μ估算能力
```

#### 2.6.6 发酵过程关键事件的自动识别与响应

BIOCore的batch-engine内置了一组**事件检测器**，自动从多参数趋势中识别发酵过程的关键时间节点：

| 事件名称 | 检测条件 | 自动响应 | 记录 |
|---------|---------|---------|------|
| **延迟期结束(Lag Phase End)** | OUR从基线开始持续上升>20%且持续>30min | 记录事件时间点，开始累积生长时间计数器 | InfluxDB event标注 |
| **DO Spike (碳源耗尽)** | DO从<20%在10min内回升至>70%，且pH同步上升>0.1 | 触发Fed-batch Phase切换；弹窗提示"碳源耗尽，建议启动补料" | SQLite state_transitions |
| **溢流代谢检测** | 碱消耗速率突增>50% 或 RQ>1.2(如有尾气数据) 或 补料速率>理论μ_max对应值 | 软报警+建议降低补料速率20% | SQLite alarms |
| **对数期结束(进入稳定期)** | OUR增速放缓至<5%/h且持续>1h；或μ估算值<0.05/h | 记录最大OD时间点；如有IPTG诱导需求则提示操作员 | InfluxDB event标注 |
| **泡沫事件** | 液位传感器(AI-6称重)突增>5%但无对应的补料泵动作 | 启动消泡剂泵P04短脉冲(5秒) | SQLite alarms |
| **pH失控** | |pH偏差|>0.5持续>5min且对应泵已满速 | 触发Hold状态→停补料→降搅拌→弹窗报警 | SQLite state_transitions + alarms |
| **温度失控** | |温度偏差|>2°C持续>3min | 蒸汽阀/冷却阀由PLC安全连锁接管；Node.js触发Abort | SQLite state_transitions + alarms |
| **变频器故障** | RS232读取到故障码非零 或 通讯超时>3秒 | PLC硬件急停搅拌；Node.js记录故障码并触发Hold | SQLite alarms |

**事件检测器在架构中的位置：**

事件检测器运行在Node.js层的`ai-analytics/event-detector.js`中，每分钟由`collector.js`触发一次。它读取最新的InfluxDB数据和软件测算值，应用上述规则表，将检测到的事件写入SQLite的`batch_events`表，并通过WebSocket推送至前端弹窗。

与PLC层的安全连锁不同，事件检测器的响应是"软性"的——它可以建议Phase切换或参数调整，但不能绕过操作员直接写入PLC（除了泡沫事件的消泡泵短脉冲，这是唯一一个自动执行的例外，因为泡沫溢出会污染排气过滤器，造成更大损失）。

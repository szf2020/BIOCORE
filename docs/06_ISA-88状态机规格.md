## 一、ISA S88层级映射

```
ISA S88 标准层级          BIOCore 实现
─────────────────────    ─────────────────
Procedure  (过程)    ←→  Recipe  (一个完整配方，如E.coli Fed-batch)
Unit Procedure (单元过程) ←→  Batch   (一次批次执行)
Operation  (操作)    ←→  Phase   (工艺操作，如加水、灭菌、发酵)
Phase      (阶段)    ←→  Step    (Phase内的控制步骤，如粗加水、精加水)
```

**关键区分：**

| 层级 | 定义位置 | 数量 | 跳转驱动 | 可配置性 |
|------|---------|------|---------|---------|
| **Phase** | 配方JSON的`phases`数组 | 每配方通常5~20个 | batch-engine按顺序推进，或由Phase完成条件触发 | 工艺工程师自由组合、排序、设置参数 |
| **Step** | batch-engine源码硬编码 | 每Phase通常3~5步（编号1~255） | **PLC传感器信号达到设定条件时自动跳步** | **固定不变**，每种Phase类型内置固定步骤序列 |

## 二、状态定义（6个状态）

```
                  interlock未通过
                  ┌─── ✗ ───→ 拒绝启动，保持Idle
                  │
      Idle ───cmd_start──→ Running ──engine_complete──→ Complete
       ▲                    │  ▲                          │
       │     runningfault   │  │                          │
       │     cmd_hold       │  │                          │
       │     safety_*       │  │                          │
       │     comm_loss ★    │  │                          │
       │                    ▼  │ cmd_restart               │
       │                  Held                             │
       │                    │                              │
       │     cmd_pause      │                              │
       │          ┌─────────┤                              │
       │          ▼         │                              │
       │       Paused ──────┘                              │
       │          │                                        │
       │          │ cmd_stop (仅Held/Paused)               │
       │          ▼                                        │
       └─ cmd_reset ◄── Stopped ◄── safety_estop(任意活动态)
```

### 2.1 各状态语义与PLC状态字

每个状态对应一个固定的**状态编码**，实时写入PLC V区供PLC程序判断：

| 状态 | 编码(VW2) | 含义 | PLC行为 |
|------|-----------|------|---------|
| **Idle** | 0 | 空闲 | 所有输出4mA/OFF，安全驻留 |
| **Running** | 1 | 正常执行 | 按当前步骤的SV执行PID |
| **Held** | 2 | 故障/人工保持/通讯断线 | 补料停，搅拌降至200rpm，温度/pH PID维持，DO级联暂停 |
| **Paused** | 3 | 计划内暂停 | 冻结所有输出，保持当前值 |
| **Stopped** | 4 | 已停机 | 所有输出归零，冷却全开 |
| **Complete** | 5 | 批次完成 | 安全降温后驻留 |

### 2.2 Held vs Paused vs Stopped

| 维度 | Held | Paused | Stopped |
|------|------|--------|---------|
| 触发来源 | runningfault / cmd_hold / safety_temp_high / **comm_loss** | cmd_pause（仅人工） | cmd_stop（仅从Held/Paused）/ safety_estop |
| 控制回路 | 温度/pH维持，补料停 | 全部维持（冻结当前状态） | **全部关闭** |
| 可恢复？ | ✓ cmd_restart → Running | ✓ cmd_unpause → Running | ✗ 只能cmd_reset → Idle |
| 发酵计时 | 继续 | 暂停 | 停止 |
| 出路 | Restart / Stop / safety_estop | Unpause / Stop / safety_estop→Held | cmd_reset → Idle |

## 三、Phase类型定义（14种标准发酵工艺操作）

| 编号 | Phase类型 | JSON值 | 含义 | 固定Step数 |
|------|----------|--------|------|-----------|
| 1 | 准备 | `prepare` | 系统初始化、阀门归位、传感器预热、称重清零 | 5 |
| 2 | 加水 | `water_fill` | 向罐内加入RO水/培养基底液 | 4 |
| 3 | 人工加料 | `manual_add` | 手动添加培养基组分、菌种等 | 4 |
| 4 | 加热 | `heating` | 升温至目标温度 | 4 |
| 5 | 搅拌 | `agitation` | 独立搅拌操作 | 3 |
| 6 | 补料 | `feeding` | 蠕动泵补料操作 | 3 |
| 7 | 控温 | `temp_control` | 温度维持/切换控制 | 3 |
| 8 | pH调节 | `ph_control` | pH校正和维持 | 3 |
| 9 | DO调节 | `do_control` | 溶氧控制 | 3 |
| 10 | 通气调节 | `aeration` | 空气流量调节 | 3 |
| 11 | 出料 | `discharge` | 排料操作 | 5 |
| 12 | 发酵 | `fermentation` | 发酵培养主体（温度+pH+DO+补料综合运行） | 3 |
| 13 | 就地清洗 | `cip` | CIP清洗循环 | 5 |
| 14 | 就地灭菌 | `sip` | SIP灭菌 | 4 |

**Phase可在配方中任意组合和重复。** 例如一个典型的E.coli Fed-batch配方可能包含：
`准备 → 加水 → 人工加料 → 就地灭菌 → 控温(降温) → 人工加料(接种) → 发酵(批次期) → 发酵(补料期) → 出料`

## 四、Step系统——每种Phase类型的固定步骤序列

### 4.1 设计原则

工艺工程师在配方中只需选择Phase类型并设定参数（温度、重量、时间等），batch-engine根据Phase类型自动执行对应的固定Step序列，Step间跳转由PLC传感器信号驱动。

### 4.2 Step完成条件类型

| 条件类型 | JSON值 | 含义 | 示例 |
|---------|--------|------|------|
| 大于等于 | `>=` | PV ≥ 设定值 | 称重 ≥ 5.0kg（加水到位） |
| 小于等于 | `<=` | PV ≤ 设定值 | 温度 ≤ 38°C（降温完成） |
| 进入死区 | `in_band` | \|PV - SV\| ≤ 死区 | 温度偏差 ≤ 0.5°C（温度稳定） |
| 持续时间 | `duration` | 步骤运行时间 ≥ 设定秒数 | 保温 ≥ 1200秒（20分钟） |
| 累积值 | `accumulated` | 累积计算值 ≥ 设定值 | F₀ ≥ 20min（灭菌值） |
| 变化量 | `delta` | PV变化量 ≥ 阈值 | 称重增量 ≥ 0.1kg（人工加料检测） |
| 组合条件 | `and` / `or` | 多条件逻辑组合 | 温度入带 AND 持续>60秒 |

### 4.3 全部14种Phase类型的固定Step序列

---

**Phase类型 1：准备（prepare）**

配方参数：无特殊参数

| Step | 名称 | 动作 | 完成条件 |
|------|------|------|---------|
| 1 | 阀门归位 | 所有阀门关闭(AO-0/1/2→4mA, Q0.4~Q0.7→OFF) | 阀门关到位反馈确认(I0.2=ON, I0.4=ON)，或duration 10s |
| 2 | 传感器自检 | 读取AI-0~AI-6，检查信号有效(>3.8mA) | 所有通道有效，或标记故障通道后duration 5s |
| 3 | 称重清零 | 记录当前称重(AI-6)作为皮重基线(tare)，写入context | duration 5s（等待读数稳定） |
| 4 | 变频器自检 | RS232读取VFD状态，故障码=0 | 应答正常且故障码=0 |
| 5 | 就绪确认 | 汇总自检结果，IL-01~IL-10全部评估 | 全部pass→Step完成；任一abort→回退Idle |

---

**Phase类型 2：加水（water_fill）**

配方参数：`target_weight_kg`, `coarse_offset_kg`(默认0.3)

| Step | 名称 | 动作 | 完成条件 |
|------|------|------|---------|
| 1 | 检查初始重量 | 读取AI-6，记录当前值 | duration 5s |
| 2 | 粗加水 | 加水阀全开 | AI-6 ≥ target - coarse_offset |
| 3 | 精加水 | 加水阀减小至20%开度 | AI-6 ≥ target |
| 4 | 关阀稳定 | 关闭加水阀 | duration 10s（等管路余水滴完） |

---

**Phase类型 3：人工加料（manual_add）**

配方参数：`prompt_message`, `expected_delta_kg`(可选), `agitation_rpm`(默认200), `timeout_min`(默认60)

| Step | 名称 | 动作 | 完成条件 |
|------|------|------|---------|
| 1 | 启动搅拌 | 启动VFD至配方设定的agitation_rpm | VFD实际转速反馈达到SV±10rpm |
| 2 | 记录基线重量 | 读取AI-6作为基线 | duration 10s |
| 3 | 等待操作员加料 | UI显示prompt_message | AI-6 delta ≥ expected_delta_kg(默认0.05) 且稳定30s；或timeout→Hold |
| 4 | 加料完成确认 | 记录实际加入量(AI-6 - 基线) | duration 5s |

---

**Phase类型 4：加热（heating）**

配方参数：`target_temp_C`, `ramp_rate_C_min`(可选)

| Step | 名称 | 动作 | 完成条件 |
|------|------|------|---------|
| 1 | 关闭冷却阀 | AO-1→4mA，等待阀门关到位 | I0.4=ON 或 duration 5s |
| 2 | 开启蒸汽阀 | 等待300ms后AO-0输出，PID_01启动 | duration 1s |
| 3 | 升温 | PID_01控制蒸汽阀 | AI-0 ≥ target_temp_C |
| 4 | 温度稳定 | PID_01继续运行 | \|AI-0 - target\| ≤ 0.5°C 持续60s |

---

**Phase类型 5：搅拌（agitation）**

配方参数：`target_rpm`

| Step | 名称 | 动作 | 完成条件 |
|------|------|------|---------|
| 1 | 启动VFD | RS232发送目标频率 | VFD应答正常 |
| 2 | 加速 | VFD按内部斜坡加速 | VFD实际转速反馈 ≥ target_rpm - 10 |
| 3 | 转速稳定 | 维持目标转速 | \|实际rpm - target\| ≤ 10 持续30s |

---

**Phase类型 6：补料（feeding）**

配方参数：`pump`(P01~P04), `mode`(constant/exponential), `target_volume_mL`或`duration_min`, `rate_mL_h`(恒速)或`initial_rate`+`mu_set`(指数)

| Step | 名称 | 动作 | 完成条件 |
|------|------|------|---------|
| 1 | 启动补料泵 | 按配方mode/rate启动泵PWM | duration 10s（确认泵启动） |
| 2 | 补料运行 | 持续补料，累积计量 | 累积补料量 ≥ target_volume_mL 或 duration ≥ duration_min |
| 3 | 停泵 | 关闭补料泵 | duration 5s |

---

**Phase类型 7：控温（temp_control）**

配方参数：`target_temp_C`, `deadband`(默认0.3)

| Step | 名称 | 动作 | 完成条件 |
|------|------|------|---------|
| 1 | 切换温控模式 | 根据PV vs SV判断加热/冷却模式，激活PID_01或PID_02 | duration 3s |
| 2 | 调温 | PID运行 | AI-0到达target ± 1°C |
| 3 | 温度稳定 | PID继续，精细调节 | \|AI-0 - target\| ≤ deadband 持续120s |

---

**Phase类型 8：pH调节（ph_control）**

配方参数：`target_pH`, `deadband`(默认0.05)

| Step | 名称 | 动作 | 完成条件 |
|------|------|------|---------|
| 1 | 启动pH PID | 激活PID_06(补碱)和PID_07(补酸) | duration 5s |
| 2 | pH调节 | 酸/碱泵PID运行 | AI-2到达target ± 0.2 |
| 3 | pH稳定 | PID继续，抗振荡延时生效 | \|AI-2 - target\| ≤ deadband 持续60s |

---

**Phase类型 9：DO调节（do_control）**

配方参数：`strategy`(active_O2/active_feed/constant_O2/constant_feed), `sv`, 策略相关参数

| Step | 名称 | 动作 | 完成条件 |
|------|------|------|---------|
| 1 | 激活DO策略 | 根据strategy配置PID_04/PID_05或补料PID | duration 5s |
| 2 | DO调节运行 | 级联PID运行或补料PID运行 | AI-3到达sv ± 5% |
| 3 | DO稳定 | 策略继续运行 | \|AI-3 - sv\| ≤ 3% 持续60s |

---

**Phase类型 10：通气调节（aeration）**

配方参数：`target_NL_min`

| Step | 名称 | 动作 | 完成条件 |
|------|------|------|---------|
| 1 | 开启进气阀 | AO-2按比例输出 | duration 3s |
| 2 | 调节流量 | PID/比例控制AO-2 | AI-5到达target ± 1 NL/min |
| 3 | 流量稳定 | 维持 | \|AI-5 - target\| ≤ 0.5 持续30s |

---

**Phase类型 11：出料（discharge）**

配方参数：`cool_to_C`(默认8), `empty_weight_kg`(空罐重)

| Step | 名称 | 动作 | 完成条件 |
|------|------|------|---------|
| 1 | 降温 | 温控切换冷却模式，PID_02运行 | AI-0 ≤ cool_to_C + 2 |
| 2 | 停搅拌 | VFD停车命令 | VFD实际转速 ≤ 5rpm |
| 3 | 停通气停泵 | AO-2→4mA，所有泵OFF | duration 5s |
| 4 | 开底阀排料 | Q0.7→ON | AI-6 ≤ empty_weight_kg(空罐) |
| 5 | 关底阀 | Q0.7→OFF | duration 3s |

---

**Phase类型 12：发酵（fermentation）**

配方参数：`duration_h`, `controls`(temperature/pH/DO完整配置), `feed_strategy`(可选), `triggers`(可选的Phase跳转条件)

| Step | 名称 | 动作 | 完成条件 |
|------|------|------|---------|
| 1 | 启动全套控制 | 激活温度PID、pH PID、DO策略、补料算法(如有) | duration 60s(全部PID入带确认) |
| 2 | 发酵运行 | 所有控制回路持续运行，数据采集 | duration ≥ duration_h；或任一trigger条件满足 |
| 3 | 发酵结束过渡 | 如有补料则停补料泵，观察DO回升 | duration 300s(5min过渡稳定) |

**trigger机制：** 发酵Phase的Step 2运行期间，StepEngine持续评估配方中定义的triggers（如DO Spike）。trigger满足时Step 2视为完成，推进至Step 3→Phase完成→进入配方中的下一个Phase。

---

**Phase类型 13：就地清洗（cip）**

配方参数：`alkali_time_min`, `acid_time_min`, `rinse_time_min`, `conductivity_target_uS`(可选), `alkali_temp_C`(默认65), `acid_temp_C`(默认50), `agitation_rpm`(默认300)

| Step | 名称 | 动作 | 完成条件 |
|------|------|------|---------|
| 1 | 碱洗 | 碱泵P01启动，搅拌300rpm，温度60°C | duration ≥ alkali_time_min × 60 |
| 2 | 水洗（碱后） | 停碱泵，开RO水冲洗 | duration ≥ rinse_time_min × 60 |
| 3 | 酸洗 | 酸泵P04启动 | duration ≥ acid_time_min × 60 |
| 4 | 水洗（酸后） | 停酸泵，开RO水冲洗 | duration ≥ rinse_time_min × 60 |
| 5 | 终点检测 | 读取电导率(如有) | 电导率 ≤ target 或 duration 60s |

---

**Phase类型 14：就地灭菌（sip）**

配方参数：`target_temp_C`(默认121), `hold_time_min`(默认20), `cool_to_C`(默认40)

| Step | 名称 | 动作 | 完成条件 |
|------|------|------|---------|
| 1 | 预排空气 | 开纯蒸汽(Q0.4)，开排气阀(Q0.6)，开排水阀(Q0.7) | AI-0 ≥ 100°C |
| 2 | 升温至灭菌温度 | 继续通蒸汽，缩小排气阀 | AI-0 ≥ target AND AI-1 ≥ target |
| 3 | 保温F₀积分 | 维持温度，F₀累积，冷点<target时F₀清零 | F₀ ≥ hold_time_min |
| 4 | 冷却 | 关纯蒸汽(Q0.4)，关蒸汽阀(AO-0→4mA)，开冷却(AO-1) | AI-0 ≤ cool_to_C |

```typescript
// batch-engine/step-engine.ts

class StepEngine {
  private steps: StepDef[];
  private currentStepIndex: number = 0;
  private stepStartedAt: number = Date.now();
  private accumulators: Map<string, number> = new Map();
  private baselines: Map<string, number> = new Map();
  private phaseParams: Record<string, any>;
  private phaseType: PhaseType;

  constructor(phaseType: PhaseType, phaseParams: Record<string, any>) {
    this.steps = STEP_DEFINITIONS[phaseType];
    this.phaseParams = phaseParams;
    this.phaseType = phaseType;
  }

  get currentStep(): number {
    return this.steps[this.currentStepIndex]?.step ?? 0;
  }

  get currentStepName(): string {
    return this.steps[this.currentStepIndex]?.name ?? '';
  }

  get totalSteps(): number {
    return this.steps.length;
  }

  /**
   * 每秒由collector.js调用，传入PLC最新过程值
   * 返回: null=留在当前Step, StepEvalResult=状态变更
   */
  evaluate(pv: ProcessSnapshot): StepEvalResult {
    const stepDef = this.steps[this.currentStepIndex];
    if (!stepDef) return { action: 'phase_complete' };

    const elapsed = (Date.now() - this.stepStartedAt) / 1000;

    // 调用该Step的固定逻辑函数
    const result = stepDef.fn(pv, this.phaseParams, elapsed, {
      accumulators: this.accumulators,
      baselines: this.baselines,
    });

    if (result.complete) {
      // 记录至step_logs
      writeStepLog({
        phaseType: this.phaseType,
        stepNumber: stepDef.step,
        stepName: stepDef.name,
        elapsedSec: elapsed,
        conditionActual: result.actualValue,
        result: 'completed',
        exitSnapshot: pv,
      });

      // 执行on_complete（如关阀）
      if (result.onCompleteActions) {
        executePLCActions(result.onCompleteActions);
      }

      // 推进
      this.currentStepIndex++;
      if (this.currentStepIndex >= this.steps.length) {
        return { action: 'phase_complete' };
      }

      this.stepStartedAt = Date.now();
      const newStep = this.steps[this.currentStepIndex];
      const entryResult = newStep.fn(pv, this.phaseParams, 0, {
        accumulators: this.accumulators,
        baselines: this.baselines,
        entry: true,
      });
      if (entryResult.entryActions) {
        executePLCActions(entryResult.entryActions);
      }

      return {
        action: 'step_advanced',
        fromStep: stepDef.step,
        toStep: newStep.step,
      };
    }

    // timeout检查
    if (result.timeout && elapsed >= result.timeout.seconds) {
      return { action: 'timeout_hold', reason: `${stepDef.name}超时` };
    }

    return { action: 'continue' };
  }
}
```

## 五、Interlock条件（10条）

在`prepare`类型Phase的Step 5（就绪确认）中执行。全部通过则推进至下一Phase；任一abort项失败→回退Idle。

| 编号 | 检查项 | 检测方式 | 失败处理 |
|------|--------|---------|---------|
| IL-01 | 传感器组在线（AI-0~AI-5全部4-20mA有效） | 各通道>3.8mA | abort |
| IL-02 | 变频器通讯正常且无故障 | RS232应答+故障码=0 | abort |
| IL-03 | 蒸汽阀关到位 | I0.2=ON | abort |
| IL-04 | 冷却阀关到位 | I0.4=ON | abort |
| IL-05 | 急停未按下 | I0.0=ON (NC) | abort |
| IL-06 | 罐盖已锁紧 | I0.5=ON | abort |
| IL-07 | PLC双向心跳正常 | VB400写入成功 + VB401读取变化(3秒内) | abort |
| IL-08 | 配方已加载且状态approved | SQLite查询 | abort |
| IL-09 | InfluxDB和SQLite可写 | 写入测试 | abort |
| IL-10 | 蒸汽/冷却水供给正常 | I0.6=ON, I0.7=ON | warn（可继续） |

## 六、RunningFault条件（10条）

Running状态下持续监测（每秒一次）。触发时Running→Held。

| 编号 | 故障条件 | 检测方式 | Hold附加动作 |
|------|---------|---------|-------------|
| RF-01 | 变频器故障码非零 | RS232读取 | 搅拌急停 |
| RF-02 | 变频器通讯中断>3s | RS232超时 | 搅拌维持最后指令 |
| RF-03 | \|罐温偏差\|>2°C持续>3min | AI-0 vs SV | 温度PID继续 |
| RF-04 | \|pH偏差\|>0.5持续>5min且泵满速 | AI-2 vs SV | 补料泵停 |
| RF-05 | DO<5%持续>5min（策略一/二） | AI-3 | 补料泵停 |
| RF-06 | 罐压>2.5bar | AI-4 | 排气阀全开，蒸汽阀关 |
| RF-07 | 任一AI通道断线(<3.6mA) | AI-0~AI-6 | 记录故障通道 |
| RF-08 | 任一AI通道饱和(>20.5mA) | AI-0~AI-6 | 记录故障通道 |
| RF-09 | 称重30min无变化但补料泵在运行 | AI-6 vs 泵状态 | 补料泵停 |
| RF-10 | 称重突增>5%且无泵动作(泡沫) | AI-6变化率 | 消泡泵P04脉冲5s |
| **RF-11** | **PC↔PLC通讯断线(VB401停滞≥3秒)** | **CommWatchdog心跳监测** | **自动Hold + 补料全停 + 搅拌降至200rpm** |

**三层故障分级：**

| 层级 | 触发条件 | 目标状态 | 可恢复 |
|------|---------|---------|--------|
| RunningFault | 变频器故障、pH失控、DO过低、传感器异常等 | →Held | ✓ Restart |
| Safety事件 | 罐温>130°C | →Held（PLC硬连锁已先行保护） | ✓ Restart |
| Safety急停 | ESTOP按钮 | →Stopped | ✗ 只能Reset |
| PLC硬连锁 | 蒸汽/冷却互锁、超温 | 不经过状态机，PLC直接动作 | 条件消除后自动恢复 |

## 七、状态×事件转移矩阵

`→X` = 转移到状态X，`—` = 忽略，`✗` = 不应出现

| 当前状态＼事件 | cmd_start | cmd_hold | cmd_restart | cmd_pause | cmd_unpause | cmd_stop | cmd_reset | engine_phase_complete | engine_all_phases_done | runningfault | safety_estop | safety_temp_high |
|---|---|---|---|---|---|---|---|---|---|---|---|---|
| **Idle** | →Running | — | — | — | — | — | — | ✗ | ✗ | — | — | — |
| **Running** | — | →Held | — | →Paused | — | — | — | (下一Phase) | →Complete | →Held | →Stopped | →Held |
| **Held** | — | — | →Running | — | — | →Stopped | — | — | — | — | →Stopped | — |
| **Paused** | — | →Held | — | — | →Running | →Stopped | — | — | — | →Held | →Stopped | →Held |
| **Stopped** | — | — | — | — | — | — | →Idle | — | — | — | — | — |
| **Complete** | — | — | — | — | — | — | →Idle | — | — | — | — | — |

**关键规则：**

1. **cmd_start仅在Idle有效**——Interlock在Running进入后第一个prepare Phase的check步骤中执行，失败→回退Idle
2. **cmd_restart仅在Held有效**——从Held恢复运行
3. **cmd_unpause仅在Paused有效**——从Paused恢复运行
4. **cmd_stop仅在Held和Paused有效**——Running不能直接Stop
5. **cmd_reset仅在Stopped和Complete有效**——回到Idle
6. **safety_estop从所有活动态→Stopped**——不可恢复
7. **safety_temp_high→Held**——PLC已保护，可恢复
8. **runningfault在Running和Paused→Held**

## 九、各状态的Entry/Exit动作

### 9.1 Idle → Running（cmd_start）

```
1. 从SQLite加载配方JSON，校验status=='approved'
2. 生成batch_id: BATCH-YYYYMMDD-NNN
3. 创建batches记录
4. 加载phases数组，currentPhaseIndex = 0
5. 初始化InfluxDB写入器，启动collector.js
6. 初始化CUSUM检测器
7. 启动RunningFault检测器
8. ▶ PLC同步写入:
   - VW2 = 1 (Running)
   - VW4 = 1 (Phase 1)
   - VW6 = Phase 1的类型编码
   - VW8 = 1 (Step 1)
9. 创建StepEngine(phase.type, phase.params)，执行Step 1
10. 记录state_transition + phase_log + step_log
```

### 9.2 Running内部——Step推进

```
StepEngine.evaluate()返回step_advanced时:
1. 记录当前Step完成时间至step_logs
2. PLC同步: VW8 = 新Step编号
3. 执行新Step的entry actions

StepEngine.evaluate()返回phase_complete时:
1. 记录Phase完成时间至phase_logs
2. currentPhaseIndex++
3. 如果所有Phase已完成 → emit('engine_all_phases_done') → Complete
4. 否则:
   - 创建新StepEngine(nextPhase.type, nextPhase.params)
   - PLC同步: VW4, VW6, VW8全部更新
   - 记录phase_log + step_log
```

### 9.3 Running → Held（cmd_hold / runningfault / safety_temp_high）

```
1. 记录holdReason + 触发来源
2. 保存Phase/Step上下文快照:
   - 当前Phase index, Step number, Step elapsed时间
   - 各泵速率, 搅拌SV, DO策略参数
3. 执行Hold动作:
   - 补料泵P02/P03: OFF
   - 碱泵P01/酸泵P04: 继续
   - 搅拌: 降至200rpm
   - 通气: 降至5 NL/min
   - 温度PID: 继续
4. 执行runningfault指定的holdActions（如有）
5. PLC同步: VW2 = 2 (Held), VW4/VW8保持
6. 写入alarms + state_transition
```

### 9.4 Held → Running（cmd_restart）

```
1. 读取holdSnapshot
2. 补料泵5分钟ramp-up恢复
3. 搅拌2分钟ramp-up恢复
4. 通气恢复至Hold前设定值
5. DO策略重新激活
6. Step计时器从中断处继续
7. PLC同步: VW2 = 1 (Running), VW4/VW8保持
8. 清空holdReason/holdSnapshot
9. 记录state_transition
```

### 9.5 Running → Paused（cmd_pause）

```
1. 暂停Step计时器
2. 所有PID维持当前输出（冻结）
3. RunningFault检测器继续运行
4. PLC同步: VW2 = 3 (Paused), VW4/VW8保持
5. 记录state_transition
```

### 9.6 Paused → Running（cmd_unpause）

```
1. 恢复Step计时器
2. PLC同步: VW2 = 1 (Running)
3. 记录state_transition
```

### 9.7 Held/Paused → Stopped（cmd_stop / safety_estop）

```
1. 判断触发来源:

   a. cmd_stop (正常停机):
      - 温控切换冷却模式，目标8°C
      - 停止所有泵
      - 关闭进气
      - 搅拌降至100rpm → 温度达标后停搅拌

   b. safety_estop (紧急停机):
      - 所有泵: OFF
      - 蒸汽阀: 关闭(AO-0→4mA)
      - 冷却阀: 全开(AO-1→20mA)
      - 进气阀: 关闭(AO-2→4mA)
      - VFD紧急停车
      - 排气阀: 开启(Q0.6→ON)
      - 声光报警: 开启(Q1.0→ON)

2. 停止RunningFault检测器
3. 停止collector.js
4. Flush InfluxDB
5. PLC同步: VW2 = 4 (Stopped), VW4=0, VW6=0, VW8=0
6. 更新batches表: ended_at, outcome
7. 写入alarms + state_transition
```

### 9.8 Running → Complete（engine_all_phases_done）

```
1. 最后一个Phase(通常是discharge)的Step已全部完成
2. 触发Ollama批次摘要生成
3. 停止RunningFault检测器
4. 停止collector.js
5. PLC同步: VW2 = 5 (Complete), VW4=0, VW6=0, VW8=0
6. 更新batches表: outcome='success', summary_text, ended_at
7. 记录state_transition
```

### 9.9 Stopped/Complete → Idle（cmd_reset）

```
1. 重置所有context
2. 声光报警关闭(Q1.0→OFF)
3. PLC同步: VW2=0, VW4=0, VW6=0, VW8=0
4. 系统就绪
```

## 十、PLC同步——V区地址映射

| 地址 | 方向 | 内容 | 说明 |
|------|------|------|------|
| VW2 | PC→PLC | 状态编码 | 0=Idle, 1=Running, 2=Held, 3=Paused, 4=Stopped, 5=Complete |
| VW4 | PC→PLC | 当前Phase编号 | 配方中Phase的序号(1~N)，非Running时为0 |
| VW6 | PC→PLC | Phase类型编码 | 1~14（见第三节），非Running时为0 |
| VW8 | PC→PLC | 当前Step编号 | Phase内Step序号(1~255)，非Running时为0 |

**PLC端使用示例：**

```

  1: // Running — 根据Phase类型和Step启用对应逻辑

      2: // 加水

        ELIF VW8 = 3 THEN  减小加水阀

      12: // 发酵
        温度/pH/DO PID全部启用
    END_CASE
  2: // Held — 补料停，搅拌降速，温度/pH维持
  3: // Paused — 冻结所有输出
  4: // Stopped — 所有输出归零
  5: // Complete — 同Stopped
END_CASE
```

## 十二、持久化与崩溃恢复

每次状态转移时将XState snapshot写入batches.state_snapshot字段。Node.js重启后从snapshot恢复状态机，若PLC心跳已超时则自动发送cmd_hold使状态机进入Held。

```typescript
// batch-engine/persistence.ts

function restoreFromCrash(): void {
  const activeBatch = db.prepare(`
    SELECT batch_id, state_snapshot 
    FROM batches 
    WHERE current_state NOT IN ('idle', 'complete', 'stopped')
    ORDER BY started_at DESC LIMIT 1
  `).get();

  if (!activeBatch) return;

  const snapshot = JSON.parse(activeBatch.state_snapshot);
  const restoredActor = createActor(batchMachine, { snapshot });
  restoredActor.start();

  if (!checkHeartbeat()) {
    restoredActor.send({ type: 'cmd_hold' });
    writeAlarm({
      batch_id: activeBatch.batch_id,
      alarm_code: 'HEARTBEAT_LOST_ON_RECOVERY',
      severity: 'warning',
      source: 'node:persistence',
      message: 'Node.js崩溃恢复后PLC心跳超时，自动Hold',
    });
  }
}
```

## 十三、状态机与PLC安全层的职责边界

| 场景 | PLC层（<1ms） | Node.js状态机（~1s） |
|------|-------------|---------------------|
| 蒸汽/冷却互锁 | ★ 硬件禁止同时开 | UI显示 |
| 罐温>130°C | ★ 蒸汽关+冷却全开 | safety_temp_high → Held |
| 急停ESTOP | ★ 所有输出归零 | safety_estop → Stopped |
| 心跳>3s | ★ PLC安全驻留 | 恢复后 → Held |
| 变频器故障 | PLC急停搅拌 | runningfault RF-01 → Held |
| **通讯断线** | **CommWatchdog检测VB401停滞** | **comm_loss RF-11 → Held (hold_reason='通讯断开')** |
| **通讯恢复** | **VB401恢复变化** | **comm_restored → 保持Held, 等待操作员cmd_restart** |
| **通讯超时** | **Hold超1小时未恢复通讯** | **safety_timeout → Stopped** |
| pH/DO失控 | PLC仅执行PID | runningfault RF-04/05 → Held |
| 传感器断线 | PLC无法检测 | runningfault RF-07 → Held |
| 启动前检查 | 无 | interlock IL-01~10 → 阻止进入Running |

## 补充：通讯断线处理规格 (v2新增)

### 双向心跳协议

PC端(Node.js)和PLC端各自运行独立的心跳程序，形成双向监控：

| 方向 | 地址 | 写入方 | 更新频率 | 监测方 | 超时阈值 |
|------|------|--------|----------|--------|----------|
| PC→PLC | VB400 | plc-driver(node-snap7) | 每秒递增(0~255) | PLC梯形图 | 3秒不变 |
| PLC→PC | VB401 | PLC梯形图(T37定时器) | 每秒递增(0~255) | CommWatchdog | 3秒不变 |

### CommWatchdog 通讯看门狗

CommWatchdog模块监听plc-driver的心跳事件，实现以下状态流转：

```
   Running ──comm_loss──→ Held(通讯断开)
                           │
                     ┌─────┤
                     │     │ comm_restored
                     │     ▼
                     │   Held(等待确认) ──cmd_restart──→ Running
                     │
                     │ safety_timeout(1h)
                     ▼
                   Stopped
```

**关键约束:**
- 通讯断线时状态机 **自动进入Held**, 不需要操作员干预
- 通讯恢复时 **不自动Restart**, 必须操作员确认后手动恢复
- Hold超过1小时未恢复通讯 → 自动转为Stopped (安全停机)
- 所有通讯事件记录到 `comm_events` 表和 `audit_logs` 表

### PLC端安全驻留

PLC梯形图中实现的独立安全逻辑 (不依赖上位机):

| VB400监测 | 动作 |
|-----------|------|
| 连续≥3秒不变 | 置位安全驻留标志V500.0 |
| V500.0=TRUE时 | 补料泵P01~P04全部OFF; 搅拌SV降至200rpm; 温度PID_01/02继续; pH PID_06/07继续; DO级联暂停(AO-2维持当前值) |
| VB400恢复变化 | 复位V500.0; PLC等待上位机重新下发SV |

#### 2.5.1 设计哲学：为什么BIOCore拒绝依赖云端AI

发酵过程通常持续数天到数周，期间实验室网络可能中断、VPN不稳定、或者所在机构的防火墙封锁了外部API访问。如果AI功能依赖云端LLM，那么恰恰在最需要AI辅助判断的紧急时刻（如凌晨3点DO突然暴跌），AI可能因为网络问题而不可用。此外，发酵过程数据涉及菌株性能、培养基配方等核心知识产权，部分企业用户对数据出境（发送到海外API服务器）有严格的合规限制。

因此BIOCore的AI架构遵循**"本地优先，云端可选"**原则：

- **所有实时功能100%本地运行**：异常检测、软测量推断、批次相似度匹配——这些在发酵过程中持续运行的功能，全部基于本地模型，零网络依赖。
- **交互式分析功能本地优先**：自然语言查询、批次报告摘要——优先使用本地部署的小型LLM（通过Ollama），在用户主动选择且网络可用时才回退到云端API。
- **云端API仅作为可选增强**：用户可以配置Anthropic/OpenAI API Key获得更强的语言理解能力，但系统不强制要求，所有核心AI功能在完全离线的环境下均可正常工作。

**关键技术组件：**

| 组件 | 用途 | 本地运行方案 | 硬件要求 |
|------|------|-----------|---------|
| 小型LLM | 自然语言查询、批次摘要、工艺建议 | Ollama + Qwen2.5-7B-Q4量化 (约4.5GB显存/内存) | 16GB RAM，无需GPU（CPU推理，约2-5 tokens/sec） |
| ML推理引擎 | 软测量模型在线推断 | onnxruntime-node + 用户训练的ONNX模型 | 极低（模型<10MB，推理<10ms） |
| DTW算法 | 批次相似度匹配、历史比对 | 纯JavaScript实现（dynamic-time-warping npm包） | 极低 |
| 统计检测 | 实时异常检测 | 滑动窗口均值/标准差 + CUSUM算法，纯JS | 极低 |

**为什么选择Ollama + Qwen2.5-7B：**

Ollama是目前本地LLM部署最成熟的方案，一条命令即可安装，暴露标准的OpenAI兼容HTTP API（localhost:11434）。Qwen2.5-7B在中英双语理解上表现优异，Q4量化后约4.5GB内存占用，在普通实验室PC（i7/Ryzen7 + 16GB RAM）上以CPU模式运行可达2-5 tokens/sec——生成一段200字的批次摘要约需30-60秒，对非实时的分析场景完全可接受。用户也可以自行替换为Llama3.1-8B、Mistral-7B或任何Ollama支持的模型。

|------|---------|-------|---------|----------|

**自然语言→Flux查询的实现细节：**

```javascript
// ai-gateway/nl-to-flux.js 核心逻辑

const SYSTEM_PROMPT = `你是BIOCore发酵数据库查询助手。
用户会用自然语言描述他们想查询的发酵数据。
你的任务是将其转换为InfluxDB Flux查询语句。

数据库Schema:
- Bucket: "fermentation"
- Measurement: "process_data"
- Tags: batch_id, reactor_id
- Fields: temperature, pH, DO, rpm, airflow, pressure, feed_rate, base_rate, acid_rate
- 采样频率: 每分钟1条记录

规则:
1. 只输出Flux查询代码，不要解释
2. 时间范围默认最近30天，除非用户指定
3. 如果涉及"最高"/"最低"等聚合，使用 max()/min()
4. 如果涉及多批次对比，使用 group(columns: ["batch_id"])

示例:
用户: "上个月OD最高的批次"
输出:
from(bucket: "fermentation")
  |> range(start: -30d)
  |> filter(fn: (r) => r._measurement == "process_data" and r._field == "DO")
  |> group(columns: ["batch_id"])
  |> max()
  |> sort(columns: ["_value"], desc: true)
  |> limit(n: 1)`;

async function naturalLanguageToFlux(userQuery) {
  // 优先尝试本地Ollama
  try {
    const response = await fetch('http://localhost:11434/v1/chat/completions', {
      method: 'POST',
      headers: { 'Content-Type': 'application/json' },
      body: JSON.stringify({
        model: 'qwen2.5:7b',
        messages: [
          { role: 'system', content: SYSTEM_PROMPT },
          { role: 'user', content: userQuery }
        ],
        temperature: 0.1  // 低温度确保查询准确性
      })
    });
    return extractFluxQuery(await response.json());
  } catch (ollamaError) {
    // Ollama不可用时，回退到云端API（如果用户配置了Key）
    if (config.cloudApiKey) {
      return callCloudLLM(userQuery);
    }
    throw new Error('本地AI服务未运行，请启动Ollama或配置云端API Key');
  }
}
```

**实时异常检测的实现细节（零AI依赖，纯统计算法）：**

```javascript
// anomaly/cusum-detector.js — 每分钟由collector.js调用

class CUSUMDetector {
  constructor(channel, options = {}) {
    this.channel = channel;         // 'temperature' | 'pH' | 'DO' | ...
    this.threshold = options.threshold || 4.0;  // 检测灵敏度(σ倍数)
    this.drift = options.drift || 1.0;          // 允许的自然漂移量
    this.windowSize = options.windowSize || 60;  // 基线计算窗口(分钟)
    this.cumSumPos = 0;
    this.cumSumNeg = 0;
  }

  // 用历史黄金批次或最近N个批次的同时间段数据建立基线
  async buildBaseline(batchIds, elapsedMinute) {
    const values = await influxClient.query(`
      from(bucket: "fermentation")
        |> range(start: -365d)
        |> filter(fn: (r) => r._field == "${this.channel}")
        |> filter(fn: (r) => ${batchIds.map(id => `r.batch_id == "${id}"`).join(' or ')})
        // 对齐到发酵经过时间而非绝对时间
    `);
    this.mean = calculateMean(values);
    this.std = calculateStd(values);
  }

  // 每分钟调用：输入当前PV值，返回是否异常
  detect(currentValue) {
    const normalized = (currentValue - this.mean) / this.std;
    this.cumSumPos = Math.max(0, this.cumSumPos + normalized - this.drift);
    this.cumSumNeg = Math.max(0, this.cumSumNeg - normalized - this.drift);

    if (this.cumSumPos > this.threshold || this.cumSumNeg > this.threshold) {
      return {
        anomaly: true,
        direction: this.cumSumPos > this.threshold ? 'high' : 'low',
        severity: Math.max(this.cumSumPos, this.cumSumNeg) / this.threshold,
        message: `${this.channel}偏离历史基线 ${normalized.toFixed(1)}σ，持续累积偏差已触发报警`
      };
    }
    return { anomaly: false };
  }
}
```

#### 2.5.4 v2.0 AI功能矩阵（本地ML模型，构建数据壁垒）

v2.0的AI功能核心是**利用用户在v1.0阶段积累的批次数据训练本地专属模型**。这些模型一旦训练完成，完全在本地运行，推理延迟<10ms，是竞品无法复制的差异化壁垒。

| 功能 | 运行位置 | AI类型 | 数据依赖 | 输出与控制权 |
|------|---------|-------|---------|-----------|
| **在线软测量（生物量推断）** | 本地ONNX Runtime | 回归模型(RandomForest/LSTM) | 需≥10个历史批次的在线数据+离线OD取样 | Dashboard显示"AI估算OD"虚线，仅供参考，不写入PLC |
| **在线软测量（底物浓度推断）** | 本地ONNX Runtime | 回归模型 | 需≥10个历史批次的在线数据+离线葡萄糖检测 | Dashboard显示"AI估算葡萄糖"虚线，不写入PLC |
| **补料策略优化建议** | 本地Ollama + 动力学拟合 | LLM + Monod模型 | 当前批次实时数据+软测量推断值 | 弹窗建议卡片，需操作员点击"采纳"才修改SV |
| **异常根因分析** | 本地Ollama | LLM推理 | 报警事件+前后30分钟多参数趋势 | 自然语言分析报告，写入audit_logs供事后复盘 |
| **实验设计推荐（贝叶斯优化）** | 本地Python子进程 | 高斯过程回归 | 所有历史批次的终点指标(产量/效价) | 推荐下一批次的温度/pH/转速/补料参数组合 |

**软测量模型的训练与部署流程：**

```
训练阶段（用户在DataExplorer中操作）：

1. 选择训练集
   用户勾选10-20个历史批次 → 系统从InfluxDB提取在线数据
   + 从SQLite offline_sample表提取离线取样数据 → 自动按时间对齐

2. 特征工程（自动）
   输入特征: temperature, pH, DO, rpm, airflow, feed_rate, elapsed_time
   目标变量: OD600 (或 glucose_g_L, product_titer)
   系统自动: 滑窗平滑、一阶差分、累积积分(如累积补料量)

3. 模型训练（本地Python子进程）
   Node.js通过child_process调用Python脚本:
   $ python3 train_soft_sensor.py \
       --input features.csv \
       --target OD600 \
       --model random_forest \
       --output models/od_sensor_v1.onnx

   训练时间: 10个批次约2-5分钟(CPU)
   输出: ONNX格式模型文件 (<10MB)

4. 验证（自动）
   留出2-3个批次做交叉验证
   计算RMSE和R²，生成拟合曲线对比图
   R² > 0.85 → 自动标记为"可用"
   R² < 0.85 → 提示用户增加训练数据或检查离线取样质量

在线推断阶段（发酵过程中自动运行）：

每分钟:
  collector.js取最新在线数据
  → soft-sensor/onnx-runner.js 加载ONNX模型
  → onnxruntime-node 推理 (<10ms)
  → 结果写入InfluxDB (measurement: "soft_sensor", field: "OD_estimated")
  → WebSocket推送至前端Dashboard
  → 前端以虚线+置信区间显示，明确标注"AI估算值"
```

**软测量值在Dashboard上的显示规范：**

| 要素 | 硬件传感器数据 | AI软测量估算值 |
|------|-------------|-------------|
| 线型 | 实线 | **虚线** |
| 颜色 | 主题色(蓝/绿) | **灰色+半透明置信区间带** |
| 标签 | "DO: 28.5%" | **"AI估算OD: 12.3 ±1.5 (R²=0.91)"** |
| 写入PLC | 可作为PID的PV | **永远不写入PLC** |
| 数据可信度 | 传感器校准状态决定 | 模型R²值+训练数据量+外推警告 |
